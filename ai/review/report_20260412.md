### Critical #1: "你好"消息触发无限催促循环，浪费Token

**所在文件**: `internal/agent/core.go` 第 474-478 行

**问题描述**:
当用户发送"你好"这类纯聊天消息时，模型的回复通常是一段问候语（如"你好！有什么可以帮你的？"），既没有 `tool_calls`，也不包含 `isTaskComplete()` 检测的完成标志（"任务已完成"/"task completed"等）。此时代码进入第 476 行的逻辑：

```go
// 如果没有工具调用且没有完成标志，可能是模型在等待更多输入
// 添加一个提示让模型继续
a.history.AddMessage(model.NewUserMessage("请继续执行任务。如果任务已完成，请使用complete_task工具标记完成。"))
```

然后 `continue` 回到循环顶部，再次调用模型。模型面对"请继续执行任务"的催促，可能再次回复一段无意义的内容，又不触发完成条件，形成 **无限催促循环**，直到耗尽 `maxIterations`（默认50次）。

**影响分析**:
- 每次迭代都调用一次LLM API，50次迭代意味着浪费大量Token和API调用
- 用户发送简单问候就导致长时间无意义的运行
- 系统提示词要求"高效执行模式"，与纯聊天场景完全不适配

**建议修复方案**:
1. 在系统提示词中增加对纯聊天消息的处理指导，如"当用户发送非任务类消息（如问候、闲聊），直接回复并调用 `complete_task` 标记完成"
2. 在 `isTaskComplete()` 中增加更多完成判断条件：如果连续N次迭代无工具调用且无任务上下文，则判定为非任务消息直接结束
3. 在外层增加"无工具调用次数计数器"，超过阈值（如3次）自动终止

### Critical #2: `verificationPassed` 标志从未被正确设置，强制校验逻辑可能永远不执行或永远执行

**所在文件**: `internal/agent/core.go` 第 45, 398, 469, 484, 525 行

**问题描述**:
`verificationPassed` 字段在 `Agent` 结构体中声明（第45行），初始化为 `false`（Go零值）。在整个 `Run` 方法中：

- 第 398 行注释说"不设置verificationPassed，让外层循环执行强制校验"
- 第 469 行注释说"不设置verificationPassed，让外层循环执行强制校验"
- 第 484 行检查 `if !a.verificationPassed`
- 第 525 行设置 `a.verificationPassed = true`

问题在于：`verificationPassed` 是 `Agent` 的实例字段，而非 `Run` 方法的局部变量。如果同一个 `Agent` 实例被多次调用 `Run`：
1. 第一次运行正常完成后，`verificationPassed` 被设为 `true`
2. 第二次运行时，即使内层循环因 `complete_task` 跳出，由于 `verificationPassed` 仍为 `true`（第484行），外层循环直接跳过强制校验，立即返回成功

**影响分析**:
- 第二次及后续任务跳过强制校验，可能导致未完成的任务被标记为完成
- 这是一个严重的状态泄漏Bug

**建议修复方案**:
```go
// 在 Run 方法开始时重置
a.verificationPassed = false

// 或改为 Run 方法的局部变量
```

### Critical #3: `SendMessage` 的 TOCTOU 竞态条件

**所在文件**: `internal/agent/core.go` 第 933-960 行

**问题描述**:
`Agent.SendMessage()` 使用"检查后执行"模式：

```go
func (a *Agent) SendMessage(content string) error {
    a.mu.RLock()
    if a.running {  // 检查
        a.mu.RUnlock()
        return fmt.Errorf("agent is already running")
    }
    a.mu.RUnlock()  // 释放锁

    go func() {     // 执行（在另一个goroutine中）
        ctx := context.Background()
        _, err := a.Run(ctx, content)  // Run内部会再次获取锁设置running=true
        ...
    }()
    return nil
}
```

在 `mu.RUnlock()` 和 `go func()` 之间存在时间窗口。如果两个 goroutine 几乎同时调用 `SendMessage`（例如前端快速双击发送按钮），两者都可能通过 `running` 检查，然后各自启动一个 `Run` goroutine。虽然 `Run` 内部有二次检查，但 `SendMessage` 已经返回 `nil`（成功），前端认为两次消息都已发送。

类似的问题也存在于 `App.SendMessage()` 中（`gui/app.go` 第 118-148 行），`isStreaming` 的检查和设置之间存在间隙（虽然间隙很小）。

**影响分析**:
- 并发调用可能导致两个 `Run` goroutine 同时运行
- `running` 标志只在 `Run` 内部被设置为 `true`，`SendMessage` 返回时尚未设置

**建议修复方案**:
```go
func (a *Agent) SendMessage(content string) error {
    a.mu.Lock()
    if a.running {
        a.mu.Unlock()
        return fmt.Errorf("agent is already running")
    }
    a.running = true  // 在锁内设置
    a.mu.Unlock()

    go func() {
        defer func() {
            a.mu.Lock()
            a.running = false
            a.mu.Unlock()
        }()
        // ... Run不再设置running
    }()
    return nil
}
```

### High #4: `App.SendMessage` 中 `isStreaming` 设置与 goroutine 错误重置不一致

**所在文件**: `internal/gui/app.go` 第 118-148 行

**问题描述**:
```go
func (a *App) SendMessage(content string) error {
    // ...
    a.isStreaming = true   // 第134行：在锁内设置
    a.mu.Unlock()

    go func() {
        if err := a.agent.SendMessage(content); err != nil {
            a.mu.Lock()
            a.isStreaming = false  // 第142行：仅在agent.SendMessage返回错误时重置
            a.mu.Unlock()
            runtime.EventsEmit(a.ctx, "stream:error", err.Error())
        }
        // 注意：如果agent.SendMessage返回nil（正常情况），isStreaming不会被这里重置
        // 依赖StreamBridge.OnTaskDone()来重置
    }()
    return nil
}
```

正常流程下，`isStreaming` 由 `StreamBridge.OnTaskDone()` 重置。但如果 `StreamBridge` 未设置、或 `OnTaskDone` 未被调用（例如 Agent 内部 panic），`isStreaming` 将永远为 `true`，用户将永远无法发送下一条消息。

**影响分析**:
- 如果 StreamBridge 出错或未设置，前端将永久卡在"正在推理"状态
- 用户必须重启应用才能恢复

**建议修复方案**:
```go
go func() {
    if err := a.agent.SendMessage(content); err != nil {
        a.mu.Lock()
        a.isStreaming = false
        a.mu.Unlock()
        runtime.EventsEmit(a.ctx, "stream:error", err.Error())
    }
    // 不依赖StreamBridge，使用defer保证最终重置
    // 或者增加超时保护
}()
```

更好的方案：在 `SendMessage` 的 goroutine 中使用 `defer` 确保 `isStreaming` 最终被重置，而不是完全依赖 `StreamBridge`。

### High #5: `StreamBridge.OnComplete` 会固化当前消息，但同一 `Run` 中多次调用会导致消息重复追加

**所在文件**: `internal/gui/stream_bridge.go` 第 151-171 行、`internal/agent/core.go` 第 729-731 行

**问题描述**:
在 Agent 的主循环中，每次 `callModel()` 完成后都会调用 `handler.OnComplete(usage)`。`OnComplete` 会将 `conv.currentMessage` 追加到 `conv.messages` 并将 `currentMessage` 设为 `nil`：

```go
func (b *StreamBridge) OnComplete(usage int) {
    // ...
    conv.messages = append(conv.messages, conv.currentMessage) // 固化消息
    conv.currentMessage = nil
    // ...
}
```

在 `Run` 的主循环中，每次迭代（最多50次）都会调用 `callModel()`，每次都会触发 `OnComplete`。这意味着：
- 第1次迭代：assistant回复 → OnComplete → 消息追加到 messages
- 第2次迭代：assistant再次回复 → OnComplete → 消息追加到 messages
- ...

这不是Bug而是一个设计问题。每次迭代都会在前端产生一条新的 assistant 消息，这可能是期望行为。但如果迭代过程中有工具调用，中间的思考过程也会被固化，可能不是期望的效果。

**影响分析**:
- 多次迭代会在前端产生多条 assistant 消息
- 用户看到的消息列表可能包含大量中间状态的回复

**建议修复方案**:
明确设计决策：要么每次迭代创建新消息，要么在迭代过程中持续更新同一消息。如果选择后者，`OnComplete` 不应固化消息，应只在 `OnTaskDone` 时固化。

### High #6: `Agent.Run` 中内层循环 `break` 后外层循环的行为不完整

**所在文件**: `internal/agent/core.go` 第 400, 471 行

**问题描述**:
当 `complete_task` 工具被检测到时（第 355-401 行），执行 `break` 跳出内层 `for` 循环。但注意此时 `result.Iterations = iteration` 在第 397 行设置了，但 `result.FinalResponse` 没有被设置（第 467 行只在无工具调用路径设置了 `FinalResponse`）。

然后外层循环第 482 行检查 `result.Status == "completed"`，执行强制校验。如果强制校验失败，状态被重置为 `"in_progress"`（第 518 行），然后 `continue` 继续外层循环。

但外层循环的下一次迭代会重新进入内层循环（`iteration` 从 1 开始），这意味着之前已消耗的迭代次数被"遗忘"了。总迭代次数可能远超 `maxIterations`，因为每次外层循环重置了内层循环的计数器。

**影响分析**:
- 总的实际LLM调用次数可能远超 `maxIterations`
- `maxOuterIterations = maxIterations + 10`（第220行）提供了上限，但与用户预期的 `maxIterations` 不一致

**建议修复方案**:
维护一个全局迭代计数器，跨内外层循环共享，而不是让内层循环每次从1开始。

### Medium #9: `GetMessagesRef()` 返回内部切片引用，破坏封装

**问题描述**:

### Medium #10: SSE 解析不够健壮，`data:` 前缀处理简单

**所在文件**: `internal/model/client.go` 第 195-196 行

**问题描述**:
```go
if strings.HasPrefix(line, "data: ") {
    data := strings.TrimPrefix(line, "data: ")
```

1. 只处理 `"data: "` （有一个空格），不处理 `"data:"`（无空格）的情况。SSE规范允许空格可有可无。
2. 非 `data:` 行被写入 `buffer`（第217-220行），但 `buffer` 从未被读取使用，是无用代码。
3. 没有处理 `event:`、`id:`、`retry:` 等标准SSE字段。
4. `bufio.Scanner` 默认有 64KB 的行长度限制（`bufio.MaxScanTokenSize`），如果单行 SSE 数据超过此限制，scanner 会报错。

**影响分析**:
- 某些LLM API返回的SSE可能不带空格，导致解析失败
- 大量JSON数据可能超过行长度限制

**建议修复方案**:
```go
if strings.HasPrefix(line, "data:") {
    data := strings.TrimPrefix(line, "data:")
    data = strings.TrimLeft(data, " ") // 兼容有无空格的情况
    // ...
```

并设置更大的 scanner buffer：
```go
scanner := bufio.NewScanner(resp.Body)
scanner.Buffer(make([]byte, 0, 1024*1024), 10*1024*1024) // 10MB
```

### Medium #11: `App.SendMessage` 在锁外调用 `runtime.EventsEmit`

**所在文件**: `internal/gui/app.go` 第 137 行

**问题描述**:
```go
a.mu.Lock()
// ... 修改状态 ...
a.mu.Unlock()

runtime.EventsEmit(a.ctx, "conversation:updated", a.GetConversationData())
```

`runtime.EventsEmit` 在 `a.mu` 锁外调用，这是正确的（避免在锁内做耗时操作）。但 `GetConversationData()` 内部会再次获取读锁。在 `Unlock` 和 `GetConversationData` 之间，另一个 goroutine 可能修改了对话数据。

虽然这种不一致通常是短暂的，但前端可能收到不一致的状态。

**影响分析**:
- 前端可能短暂显示不一致的数据
- 实际影响较小，因为前端会持续收到更新事件

**建议修复方案**:
在锁内准备 `conversationData`，然后在锁外 emit：
```go
a.mu.Lock()
// ... 修改状态 ...
data := a.buildConversationDataLocked() // 不获取锁的内部方法
a.mu.Unlock()
runtime.EventsEmit(a.ctx, "conversation:updated", data)
```

### Medium #12: HTTP Client 超时设置过长

**所在文件**: `internal/model/client.go` 第 36 行

**问题描述**:
```go
httpClient: &http.Client{
    Timeout: 300 * time.Minute, // 5小时
},
```

5小时的HTTP超时设置过长。即使是流式响应，连接也应该有合理的超时。如果LLM服务器崩溃但不关闭连接，客户端会等待5小时才超时。

而且这个 `Timeout` 设置会影响到整个HTTP请求（包括连接、发送、读取），对于流式响应来说，这意味着如果在5小时内没有完成整个流的读取，才会超时。

**影响分析**:
- 在LLM服务器异常时，用户需要等待极长时间才能得到超时错误
- 资源（goroutine、内存）被长时间占用

**建议修复方案**:
使用更合理的超时（如10分钟），或为流式和非流式请求使用不同的HTTP客户端配置。

### Medium #15: Token估算过于粗略，中文场景严重偏低

**所在文件**: `internal/model/client.go` 第 270-280 行、`internal/agent/core.go` 第 721-726 行

**问题描述**:
```go
// 简单估算：平均每4个字符约1个token
totalChars := len(h.messages)
return totalChars / 4
```

`len(string)` 在 Go 中返回字节数而非字符数。对于中文，每个 UTF-8 字符占 3 个字节。一段 100 个中文字符的文本，`len()` 返回 300，按 4 字节/token 估算为 75 token。但实际中文 token 数可能只有 100-150（取决于分词器）。

更严重的是，这个估算用于 `IsContextOverflow` 和 `Truncate`，如果估算偏低，可能导致实际超出上下文限制而被LLM API拒绝。

**影响分析**:
- 中文场景下 Token 估算严重偏高（因为 `len()` 返回字节数）
- 可能导致过早触发历史截断，丢失重要上下文
- 但也有可能在英文场景下估算偏低

**建议修复方案**:
使用 `utf8.RuneCountInString()` 计算字符数，或使用更准确的 token 估算公式：
```go
runeCount := utf8.RuneCountInString(msg.Content)
totalChars += runeCount
```

### Low #17: `extractFilePaths` 中的正则表达式匹配过于宽泛

**所在文件**: `internal/agent/reviewer.go` 第 980-984 行

### Low #18: `truncateString` 按 UTF-8 字节截断，可能截断中文字符

**所在文件**: `internal/agent/reviewer.go` 第 1006-1011 行、`cmd/agentplus/main.go` 第 548-553 行

**问题描述**:
```go
func truncateString(s string, maxLen int) string {
    if len(s) <= maxLen {
        return s
    }
    return s[:maxLen] + "..."
}
```

`len(s)` 和 `s[:maxLen]` 操作的是字节，不是字符。如果 `maxLen=100` 且字符串包含中文，`s[:100]` 可能截断一个 UTF-8 字符的中间字节，产生无效的 UTF-8 字符串。

**影响分析**:
- 日志和错误消息中可能出现乱码
- 可能导致 JSON 序列化问题

**建议修复方案**:
```go
func truncateString(s string, maxLen int) string {
    runes := []rune(s)
    if len(runes) <= maxLen {
        return s
    }
    return string(runes[:maxLen]) + "..."
}
```

a.mu.Lock()
if a.isStreaming {

### 核心风险

1. **最严重的问题**是 "你好" 消息触发无限循环（Critical #1）。对于桌面应用，用户第一次使用时发送"你好"就可能导致应用看似卡死（一直在后台调用API），用户体验极差。

2. **第二严重**是 `verificationPassed` 状态泄漏（Critical #2），导致 Agent 复用时校验被跳过。

3. **第三严重**是 TOCTOU 竞态（Critical #3），虽然触发概率低，但在并发场景下可能导致不可预测的行为。

### 建议优先修复顺序

1. Critical #1: "你好"消息的无限循环 → 添加非任务消息检测和提前退出机制
2. Critical #2 + Medium #16: `verificationPassed` 状态泄漏 → 在 `Run` 开始时重置
3. Critical #3: `SendMessage` TOCTOU → 合并检查和设置为原子操作
4. High #4: `isStreaming` 卡死保护 → 添加 fallback 重置机制
5. High #7: 校验器实现 → 实现真正的校验逻辑或移除假校验器
