## 2026-03-10: AI 并发控制和会话压缩配置需求

### 用户需求概述
用户希望在服务端配置文件中添加以下两个关键功能配置：

### 功能需求 1: 自动压缩会话阈值
- **触发条件**: 基于 LM Studio 返回的模型最大 token 长度
- **配置方式**: 设置一个百分比阈值
- **功能描述**: 当会话 token 使用量达到模型最大 token 的指定百分比时，自动触发会话压缩

### 功能需求 2: AI 模型请求并发控制
- **并发限制**: 限制对 LM Studio 同时发起的 AI 模型请求数量
- **队列管理**: 超过并发限制的请求进入队列排队等待
- **调度策略**: 先到先得（FIFO）

### 技术约束
- 使用 HOCON 配置文件格式
- 与现有配置系统保持一致
- 需要可行性验证

### 配置示例
```hocon
aiService {
    # 会话压缩配置
    sessionCompression {
        enabled = true
        thresholdPercent = 0.8  # 80% 时触发压缩
    }
    
    # 并发控制配置
    concurrency {
        maxConcurrentRequests = 3  # 最多 3 个并发请求
        queueEnabled = true        # 启用队列
        queueMode = "fifo"         # 先到先得
    }
}
```

## 2026-03-10: AI 角色和会话系统详细需求

### 用户需求概述
用户详细描述了 AI 角色系统和会话系统的设计细节，包括角色工作区隔离、群聊机制、疲劳值系统、实时打断、技能管理、模型要求、记忆系统等核心功能。

### 核心功能需求

#### 1. AI 角色工作区隔离
- **隔离方式**: 使用文件系统隔离，每个 AI 角色有独立的目录
- **独立内容**:
  - 独立的系统提示词
  - 独立的行为准则和约束
  - 独立的记忆（通过 mem0）
  - 可能使用不同的模型
- **存储结构**: 每个角色在文件系统中有一个专属目录

#### 2. AI 群聊调度机制
- **调度方式**: 每次用户消息都询问所有 AI 角色是否需要回应
- **优化方案**: 设置一个调度器 AI 角色，统管群聊信息，判断当前话题与哪些角色相关
- **频率控制**: 通过配置文件控制群聊频率

#### 3. 疲劳值系统
- **计算方式**: 时间衰减的动态值
  - 基础：2 小时内不超过 200 条消息
  - 疲劳值随发言次数增加而增加
  - 疲劳值随时间衰减
- **持久化**: 疲劳值需要持久化存储
- **影响方式**: 通过系统提示词控制 AI 回应积极性
  - 疲劳值越高，AI 越期望结束会话
  - 疲劳值达到阈值后，AI 不再回应

#### 4. 实时打断功能
- **打断范围**: 用户发送新消息时，打断所有 AI 的当前处理
- **打断内容**:
  - 正在进行的模型请求
  - 正在执行的工具调用
  - 正在运行的程序/脚本
- **回复处理**: 已生成的部分回复丢弃，并在会话中撤回
- **AI 间不打断**: AI 之间的对话不打断彼此，只有用户消息能打断 AI

#### 5. 技能黑白名单系统
- **黑白名单关系**: 两者结合，白名单优先
  - 先检查白名单（如果在白名单中，允许使用）
  - 再检查黑名单（如果在黑名单中，禁止使用）
- **学习限制**: 技能学习不阻止，但学习后受黑白名单限制无法使用
- **权限控制**: 只有用户可以让 AI 学习技能，AI 角色之间不能让彼此学习技能

#### 6. 模型选择限制
- **硬性要求**:
  - 上下文长度至少支持 100K tokens
  - 必须支持图片（多模态）
  - 必须支持工具调用
- **限制方式**: 在系统设置和 AI 角色配置中，不满足要求的模型不允许选择

#### 7. 记忆系统集成
- **使用场景**: 两者结合
  - 会话压缩时，将压缩内容存储到 mem0
  - 每次对话前，从 mem0 检索相关记忆并注入上下文
- **记忆隔离**: 每个 AI 角色的记忆完全隔离（通过不同的 user_id 或 session_id）
- **群聊记忆**: 群聊中的记忆共享（可能有独立的群聊记忆空间）

### 技术栈调整

#### 重大变更
- **后端技术栈**: 从 Kotlin Native 改为 **Kotlin/JVM**
- **变更原因**: 避免项目实施过于复杂，利用 JVM 生态系统的丰富库支持

#### 技术约束
- 使用 Kotlin/JVM 进行服务端开发
- 仍然使用 Ktor 作为网络框架（支持 JVM）
- 仍然使用 Kotlin 协程进行异步处理
- 可以使用 JVM 生态系统的成熟库（如线程池、并发工具等）
- 前端客户端仍然使用 Kotlin Multiplatform（Desktop/Android/iOS/Web）

## 2026-03-11: AI 角色基础工具需求

### 用户需求概述
用户描述了每个AI角色应该拥有的基础工具能力，这些工具是角色执行任务的底层能力支撑。

### 基础工具清单

#### 1. 读取工具
- **功能**: 读取文本内容，可以指定行数范围
- **限制**: 仅限工作区内的文件
- **用途**: 查看文件内容、配置文件、代码等

#### 2. 查找工具
- **功能**: 在工作区内查找文件，可使用通配符
- **用途**: 定位文件、搜索资源等

#### 3. 修改工具
- **功能**: 修改工作区内的文件
- **使用方式**: 提供修改的原文和修改后的内容，原文用于程序定位修改的部分，然后使用修改后的内容替换
- **要求**: 提供的原文应该是部分文本，如果定位到多个，工具会告诉角色需要更多的上下文
- **用途**: 编辑文件、更新配置、修改代码等

#### 4. 删除工具
- **功能**: 将工作区内的文件或目录移动到项目的回收站目录
- **命名规则**: 文件名或目录名改为`角色id-删除时间unix时间戳-原文件名/目录名`
- **目的**: 避免数据被硬删除，保留恢复可能性
- **用途**: 清理不需要的文件、临时文件等

#### 5. 浏览器工具
- **功能**: 每个角色都有自己独立的浏览器用户数据，通过CDP协议操作
- **注意事项**: 不同角色可能在同时使用浏览器，需要考虑是否能够同时运行，或者使用其它无头浏览器方案
- **用途**: 网页浏览、信息获取、自动化操作等

#### 6. 技能创建和修改工具
- **功能**: 角色可以在自己的工作区内创建或修改自己的独有技能
- **生效方式**: 变动应立即对此角色生效
- **实现语言**: 仅允许使用Python作为程序逻辑实现
- **用途**: 扩展角色能力、自定义工具等

#### 7. 记忆查找工具
- **功能**: 主动从mem0中检索自己的记忆
- **用途**: 回忆历史信息、获取上下文等

#### 8. 自我认知和行为准则修改工具
- **功能**: 当角色觉得需要时，查看并修改自己的认知和性格，以及行为准则
- **用途**: 自我调整、优化行为模式等

### 技术约束
- 所有工具操作都限制在角色的工作区内
- 浏览器工具需要考虑多角色并发使用的问题
- 技能创建仅支持Python语言
- 删除操作采用软删除机制，保留恢复可能

### 可行性测试要求
- 测试工作区隔离机制
- 测试文件修改的定位和替换准确性
- 测试浏览器CDP协议在多角色场景下的并发能力

## 2026-03-12: 类 OAuth2.0 认证系统需求

### 用户需求概述
用户希望客户端和服务端使用类似 OAuth2.0 的方式进行认证，采用方案 A（简化版 Token 交换机制）。

### 核心功能需求

#### 1. 服务端预设 Token
- **生成方式**：服务端首次启动时自动生成
- **更新策略**：定期更换，每天一次
- **使用次数**：可以无限次使用，用于换取专属 Token
- **用途**：作为"引导 Token"，客户端首次认证时使用

#### 2. 客户端专属 Token
- **获取方式**：客户端使用预设 Token 向服务端索取
- **特性**：永不过期，每个客户端独享
- **持久化**：客户端在文件中持久化存储 Token
- **丢失处理**：Token 丢失后，需要用户重新输入预设 Token 获取新的 Token

#### 3. 服务端 Token 管理
- **持久化存储**：服务端将客户端的专属 Token 持久化到文件中
- **启动加载**：服务端启动时加载已授权的客户端 Token
- **管理功能**：支持查看、撤销已授权的客户端 Token

#### 4. 认证协议细节
- **认证格式**：使用 HTTP Header 中的 `Authorization: Bearer <token>` 格式
- **无需刷新**：Token 永不过期，不需要刷新机制
- **多端支持**：所有平台使用同一套认证机制

#### 5. 多设备支持
- **设备隔离**：同一客户端在不同设备上运行，各自使用不同的 Token
- **设备标识**：每个设备生成独立的客户端标识

### 认证流程

```
1. 服务端启动
   ├─ 生成/加载预设 Token（每天更换）
   └─ 加载已授权的客户端 Tokens（从文件）

2. 客户端首次认证
   ├─ 用户输入预设 Token
   ├─ 客户端使用预设 Token 请求服务端
   ├─ 服务端验证预设 Token 正确
   ├─ 服务端生成专属 Token 并返回
   ├─ 服务端保存专属 Token 到授权文件
   └─ 客户端持久化存储专属 Token

3. 客户端后续请求
   ├─ 客户端使用专属 Token 进行认证
   └─ 服务端验证专属 Token 在授权列表中
```

### 技术约束
- 使用 Ktor 作为网络框架
- 使用 Kotlin/JVM 进行服务端开发
- Token 使用安全的随机数生成算法
- 支持跨平台客户端（Desktop/Android/iOS/Web）
